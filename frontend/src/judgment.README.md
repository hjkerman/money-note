# judgment 모듈

`judgment.ts`는 money-note의 판단 모듈, 다시 말해 고나리질 모듈이다.
가계부 데이터 자체를 저장하거나 계산하는 핵심 장부 로직은 아니지만, 사용내역을 어떻게 분류하고 어떤 말투로 평가할지를 결정하므로 앱의 성격을 좌우한다.

## 이 모듈이 맡는 일

- 당월 지출 분류 라벨 표시
- 청구 항목 자동 분류
- 소비 통계 카드의 문구
- 요약 상단의 예산심사위원회 한 줄 평
- 가족카드 한도 감시 문구
- 이번달 결제의 파산심사위원회 문구

## 문구 변경 원칙

판단 문구는 완전한 무작위로 바뀌지 않는다.

- 새로고침만 했을 때는 같은 장부 상태에 같은 문구를 보여준다.
- 상단 `예산심사위원회 한 줄 평`은 현금, 카드, 청구, 타인정산, 동결 중 무엇이든 새 항목이 추가되면 다른 후보 문구로 순환한다.
- 가족카드 감시와 파산심사위원회는 자기 판단에 쓰는 금액이나 날짜가 바뀌었을 때 문구가 달라진다.
- 같은 위험 등급 안에서도 원인이 다르면 다른 문구를 사용한다.

이 원칙은 문구가 광고 배너처럼 무작위로 깜빡이지 않고, 실제 장부를 읽고 내린 판결처럼 보이게 하기 위한 것이다.

## 분류 체계

현재 분류는 세 가지다.

- `essential`: `안 썼으면 큰일 났을 돈`
- `questionable`: `꼭 써야 했을까...?`
- `null`: `미분류`

당월 지출은 사용자가 직접 분류한다.
청구 항목은 사용자가 조작할 여지를 줄이기 위해 `classifyClaimPanel()`이 자동 분류한다.

## 청구 자동 분류 기준

`classifyClaimPanel()`은 아래 순서로 판단한다.

1. 2,000원 이하의 소액 청구는 `questionable`
2. 의료, 대출, 교통, LPG, 가스, 유류, 하이패스 등은 `essential`
3. 커피, 간식, 게임, 취미성 키워드는 `questionable`
4. 나머지는 `미분류`

기준 금액은 `SMALL_CLAIM_LIMIT`에서 바꾼다.
키워드는 `ESSENTIAL_CLAIM_PATTERN`, `QUESTIONABLE_CLAIM_PATTERN`에서 바꾼다.

## 말투 바꾸기

말투를 바꾸고 싶다면 대체로 아래 함수만 보면 된다.

- `categoryLabel()`: 분류 이름
- `spendingStatTones()`: 통계 카드 제목과 설명
- `budgetCommitteeTone()`: 장부 전체 변화에 반응하는 예산심사위원회 한 줄 평
- `creditUsageTone()`: 가족카드 한도 감시 문구
- `paymentPressureTone()`: 결제일과 미결제액을 평가하는 파산심사위원회 문구
- `chooseJudgment()`: 같은 입력에는 같은 후보 문구를 고르는 결정적 선택기

말투의 목표는 비속어 없이, 고급스럽게 스스로를 조금 멕이는 것이다.
앱은 혼내는 척하지만 사용자를 미워하지 않는다.

## 예산심사위원회와 파산심사위원회

두 위원회는 향후 계속 개선할 정책 모듈이다. 장부 저장이나 결제 계산과 판단 문구를 섞지 않는다.

### 예산심사위원회

현재 `budgetCommitteeTone()`, `spendingStatTones()`와 소비 분류 문구가 담당한다.

- 이미 발생한 소비를 성격별로 평가한다.
- `essential`, `questionable`, `미분류` 통계를 읽기 쉬운 말로 바꾼다.
- 당월 지출, 현금흐름, 청구, 타인정산, 동결의 건수와 금액을 함께 보고 한 줄 총평을 만든다.
- 새 항목이 추가되면 같은 판단 구간에서도 다음 후보 문구로 순환한다.
- 판단 기준을 변경해도 원래 사용내역과 금액은 바뀌지 않는다.

### 파산심사위원회

`paymentPressureTone(remainingAmount, daysUntilDue, referenceLiquidity)`가 담당한다.

입력:

- `remainingAmount`: 기록상 남은 결제금액
- `daysUntilDue`: 익월 14일까지 남은 날짜. 음수면 결제일 경과
- `referenceLiquidity`: 해당 결제월의 주 수입 합계. 주 수입이 없으면 기본 심사 기준액

현재 판단 축:

1. 남은 결제금액이 0원이면 심사 종료
2. 결제일이 지났으면 가장 강한 경고
3. 결제일 당일은 별도 긴급 문구
4. 미결제액이 기준 수입의 200% 이상이면 날짜가 남아도 위험
5. 결제일까지 2일 이하이거나 미결제액이 기준 수입의 150% 이상이면 위험
6. 결제일까지 5일 이하이면서 미결제액이 기준 수입의 75% 이상이면 복합 경고
7. 미결제액이 기준 수입 이상이면 금액 경고
8. 결제일까지 5일 이하이면 일정 경고
9. 미결제액이 기준 수입의 50%, 75%를 넘는 구간을 각각 구분
10. 나머지는 통제 가능한 상태

각 구간에는 여러 후보 문구가 있다. `remainingAmount`, `daysUntilDue`, `referenceLiquidity`가 같으면 같은 문구를 돌려주고, 값이 달라지면 같은 위험 등급 안에서도 다른 문구가 선택될 수 있다.

문구와 임계값을 바꿀 때는 결제 계산 로직을 수정하지 않고 `paymentPressureTone()`만 고친다.

## 백엔드 판단 모듈

공개 공유 페이지의 문구는 백엔드의 `backend/app/services/judgment.py`가 담당한다.
어머니께 보여드리는 청구 문구, 타인정산 공유 문구, 전월~당월 가계부 평가 문구는 그 파일에서 관리한다.

백엔드 판단은 다음 세부 정보를 사용한다.

- 청구: 의료/치과/교통 키워드 비중, 2,000원 이하 소액 청구 수, 건수, 합계, 최고액
- 의료 청구 기준선: 의료 2건 이하는 정기 유지보수인 안 아픈 달, 3건은 정기 진료 외에 한 번 아팠던 달, 4건 이상은 집에서 걱정할 만큼 많이 아픈 달로 판단
- 타인정산: 본인 카드 사용액을 합친 한도 사용률, 건수, 최고액 집중도
- 가계부 평가: 카드 기록 총액, 현금흐름, 성찰 대상 소비 건수

`stable_choice()`는 같은 공유 데이터에 같은 문구를 돌려준다. 가족이 링크를 새로 열 때마다 가계부의 증언이 바뀌는 일은 없다.

프론트와 백엔드에 판단 모듈이 나뉘어 있는 이유는 다음과 같다.

- 프론트: 앱 안에서 즉시 보이는 분류와 통계
- 백엔드: 인증 없이 열리는 공유 페이지 문구

공개 페이지에는 계좌번호, 송금 링크, 개인정보를 넣지 않는다.
금액과 항목은 보여줄 수 있지만, 개인정보는 돈보다 오래 문제를 만든다.
