# 알려진 이슈

## Judgment 전면 개선 필요

Judgment 문구는 현재 대부분 서버에서 완성된 문장으로 내려온다.

- 본체 웹앱의 예산심사위원회, 카드 한도 감시, 파산심사위원회 문구는 `backend/app/services/judgment/` 패키지에서 생성한다.
- 청구/가족카드 공유 페이지 상단 문구도 서버의 `shared_panel_subtitle()` 계열에서 생성한다.
- 프론트엔드는 대부분 서버가 준 `message`, `caption`, `subtitle`을 표시한다.
- 소비 통계 영역은 서버가 준 분류별 제목/문구에 프론트에서 계산한 금액을 붙여 보여준다.

따라서 현재 상태에서는 "UI에서 문구를 조립해서 생기는 결합도 문제"는 크지 않다. 다만 판단 모듈 자체의 정책과 표현은 대대적으로 재검토해야 한다.

개선이 필요한 예:

- 청구 보고서나 공유 문구가 불필요하게 정확한 액수, 사적인 소비 평가, 내부 장부의 판단을 노출하지 않도록 해야 한다.
- 가족에게 보여주는 문구와 본인만 보는 문구의 경계를 더 명확히 나눠야 한다.
- "재미있는 고나리질"과 "공유 가능한 회계 안내"를 같은 기준으로 만들지 말아야 한다.
- claim, family_card, ledger_entries 각각의 도메인 차이를 문구 생성 기준에 더 강하게 반영해야 한다.
- 판단 문구의 출력 대상별 정책을 문서화해야 한다. 예: 본인 화면, 청구 공유 페이지, 가족카드 공유 페이지.
- 향후 judgment 모듈은 문구 후보만 늘리는 방식이 아니라, 공개 범위와 민감도에 따른 레이어를 가져야 한다.

문구 후보 파일의 관리 방식은 [Judgment 문구 관리](judgment-messages.md)를 따른다.

## 주의사항

- Snapshot restore는 장부 운용 데이터를 교체하는 위험 작업이다. 사용자 계정, 세션, 관리 로그, 공유 PIN 해시는 포함하거나 복원하지 않는다.
- Snapshot restore는 manifest 검증, 임시 DB dry-run, mandatory `pre_restore` 생성과 검증을 통과해야 실제 운영 DB를 수정한다.
- `pre_restore`는 설정 모달에서 목록 조회, 삭제, 되돌리기를 할 수 있으며, filename whitelist와 경로 검증으로 `snapshot-backups` 밖의 파일 접근을 막는다.
- Snapshot restore는 manifest 검증 후 현재 서버 스키마에 없는 컬럼을 무시한다. 새 필드가 생겨도 구버전 백업을 가능한 한 복원하기 위한 정책이며, 민감 설정이나 manifest 불일치까지 허용한다는 뜻은 아니다.

## 알림 수집 후속 작업

- 고속도로 통행료+ `com.ex.hipass_app`은 하이패스 이용내역 알림을 제공하지만 정확한 알림 본문 규격이 공개되어 있지 않다.
- 현재 모바일 앱은 해당 패키지 알림을 출처별 최근 30건 원문 로그로만 보관한다. 파싱, 후보 생성, 서버 전송은 의도적으로 비활성화했다.
- 실제 원문 표본에서 날짜, 금액, 진입·진출 영업소가 안정적으로 확인되면 우리카드 파서와 분리된 하이패스 파서를 추가한다.
- 고속도로 통행료+ 알림이 실제 환경에서 제공되지 않거나 안정적으로 수집되지 않으면 다음 실험 순서를 검토한다.
  1. 내비게이션 앱이 알림, 공유 기능 등 공개적으로 노출하는 예상 통행료 정보를 수집한다.
  2. 사용자가 명시적으로 시작한 운전 모드에서 하이패스 단말기 또는 내비게이션의 통행료 안내 음성을 짧게 관측하고, 기기 안에서 필요한 숫자만 추출한 뒤 원음은 폐기한다.
- 음성 관측은 상시 녹음으로 구현하지 않는다. Android의 백그라운드 마이크 제한, 배터리 사용량, 동승자 개인정보, 차량 소음에 따른 오인식 때문에 fallback이자 experimental idea로만 둔다.
- 후불 하이패스카드나 차량 하이패스 단말기를 직접 읽는 방식은 폐쇄형 통신 규격, 별도 하드웨어, 카드 보안과 차량 장치 개조 가능성 때문에 현재 구현 후보에서 제외한다.
- 사용자의 실기기에서는 모바일티머니 `com.lgt.tmoney`가 삼성월렛 모바일티머니의 결제 금액과 잔액 알림을 제공하는 것을 확인했다.
- 현재 모바일티머니 알림도 출처별 최근 30건 원문 로그로만 보관한다. 실제 문장 형식과 승차·하차별 필드가 충분히 관측되기 전에는 파싱, 후보 생성, 서버 전송을 하지 않는다.
- 표본이 충분하면 우리카드·하이패스 파서와 분리된 모바일티머니 파서를 추가한다. 결과는 자동 등록이 아니라 사용자가 본인 사용 또는 청구 사용과 금액을 확인하는 후보로 만든다.

## 해결됨

- 정기결제 등록 후 200 OK인데 화면에 반영되지 않던 문제는 `confirmed_month IS NULL` 조회 조건 문제였고, 현재 테스트로 회귀를 막는다.
- 모바일 Judgment가 상태 탭 안쪽에서만 보이던 문제는 입력 탭의 `오늘의 예산심사위원회` 요약 카드와 상태 탭의 전체 Judgment 카드로 재배치했다. Judgment 생성 로직과 API 응답은 변경하지 않았다.
