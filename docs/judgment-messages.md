# Judgment 문구 관리

이 문서는 에이전트가 Money Note의 Judgment 문구를 수정할 때 따라야 하는 작업 지침이다.

Judgment는 단순 안내 문구가 아니라 Money Note의 캐릭터다. 예산심사위원회, 파산심사위원회, 가족 공유 화면의 생활 드립은 사용자가 엑셀 대신 이 앱을 쓰는 핵심 이유 중 하나다. 따라서 문구를 고칠 때는 기능 수정처럼 조심스럽게 다룬다.

Judgment 문구는 서버가 완성 문장으로 만들어 내려준다.

프론트엔드는 문구를 조립하지 않고, 서버 응답의 `message`, `caption`, `subtitle`을 표시한다.

## 파일 위치

문구 후보는 아래 YAML 파일에 둔다.

```text
backend/app/services/judgment/messages/
  insight.yaml
  claim.yaml
  family_card.yaml
```

역할:

- `insight.yaml`: 본인 앱의 예산심사위원회, 파산심사위원회, 카드 한도 감시 문구
- `claim.yaml`: 어머니가 보는 청구 공유 화면 문구
- `family_card.yaml`: 여동생이 보는 가족카드 공유 화면 문구

판단 로직은 아래 Python 파일에 둔다.

```text
backend/app/services/judgment/
  common.py
  features.py
  insight.py
  claim.py
  family_card.py
  categories.py
```

문구만 바꾸는 작업이라면 Python 파일을 수정하지 않는다. 분기 조건이나 feature 계산을 바꿔야 할 때만 Python 파일을 수정한다.

## 수정 원칙

- 본인 앱, 청구 공유, 가족카드 공유 문구 pool을 섞지 않는다.
- 문구만 추가하거나 바꾸는 작업은 가능하면 YAML만 수정한다.
- 판단 기준, API 응답 key, DB 값은 문구 파일에서 바꾸지 않는다.
- 각 문구는 UI에서 한눈에 읽히는 길이를 유지한다.
- 문구는 서버에서 완성 문장으로 내려준다. 프론트엔드에서 조립하지 않는다.
- 금액, 비율, 날짜 같은 동적 값을 문구에 새로 넣고 싶다면 먼저 노출 정책을 확인한다.
- 문구 후보를 추가할 때는 같은 뜻을 단어만 바꿔 늘리지 말고, 관찰 포인트가 조금씩 다른 후보를 넣는다.
- 비속어는 쓰지 않는다. 사용자를 놀리더라도 고급스럽게, 스스로를 멕이는 톤으로 쓴다.
- 지나치게 추상적인 드립만 늘리지 않는다. 사용자가 지금 어떤 재정 상태인지 바로 이해할 수 있어야 한다.

## 작업 절차

에이전트는 Judgment 문구 작업 전에 다음 순서로 확인한다.

1. 이 문서를 읽는다.
2. 수정 대상 YAML 파일을 읽는다.
3. 관련 판단 로직 파일을 읽는다.
   - 본인 앱: `insight.py`
   - 청구 공유: `claim.py`
   - 가족카드 공유: `family_card.py`
   - 통계/분류: `categories.py`
4. 기존 테스트에서 어떤 문구 안전장치를 검증하는지 확인한다.
5. 문구만 바꾸는 작업이면 YAML만 수정한다.
6. 로직을 바꾸는 경우에는 API 응답 shape가 바뀌지 않는지 확인한다.

문구 추가 작업의 기본 단위는 "분기별 후보 수 증가"다. 한 분기에 후보를 1개만 추가하기보다, 노출 빈도가 높은 분기에는 여러 후보를 한 번에 보강한다.

## 절대 하지 말 것

- 프론트엔드에서 Judgment 문구를 새로 조립하지 않는다.
- 본인 앱 문구를 공유 화면에 재사용하지 않는다.
- 공유 화면 문구를 본인 앱 문구에 재사용하지 않는다.
- API 응답 key 이름을 바꾸지 않는다.
- `app_judgment`, `shared_panel_subtitle`, `claim_ledger_note`, `spending_stat_tones`, `category_label`의 public interface를 깨지 않는다.
- 가족카드를 핵심 원장 도메인처럼 취급하지 않는다.
- 청구와 가족카드를 "사용자가 직접 소비한 돈"처럼 표현하지 않는다.
- 정확한 개인 카드 총액이나 현금흐름 금액을 가족 공유 화면에 노출하지 않는다.

## 공유 화면 주의사항

청구 공유 화면:

- 개인 카드 총액과 현금흐름의 정확한 금액을 노출하지 않는다.
- `questionable` 같은 내부 분류명을 직접 노출하지 않는다.
- 어머니가 바로 이해할 수 있는 생활 관찰 문구를 우선한다.
- 청구는 집안 생활비/가족 관련 회수 예정 내역이다. 사용자의 개인 소비 통계처럼 쓰지 않는다.
- 건강, 병원, 약국, 정신건강, 상담, 정기 진료 언급은 숨기지 않아도 된다.
- 다만 어머니가 불필요하게 걱정하지 않도록, 걱정 유발보다 생활 관찰과 정리의 톤을 우선한다.

가족카드 공유 화면:

- `오빠`라는 단어를 쓰지 않는다.
- 여동생을 과하게 비난하거나, 갚지 않을 사람처럼 몰아가지 않는다.
- `명의자`, `카드 주인`, `가족카드`, `한도`, `신용`, `결제일` 중심으로 쓴다.
- 가족카드는 여동생이 쓰고 나중에 갚는 회수 예정 내역이다.
- 사용액이 커졌을 때는 "안 갚을 것"이 아니라 "명의자의 한도와 결제 부담이 먼저 반응한다"는 식으로 표현한다.
- 가족카드와 명의자 사용분이 함께 커진 경우에는 책임을 한쪽에만 몰지 않는다.

## 본인 앱 문구 톤

본인 앱은 사용자가 혼자 보는 화면이다. 가장 자유롭게 놀릴 수 있지만, 그래도 재정 판단이 먼저다.

허용되는 세계관:

- 예산심사위원회
- 파산심사위원회
- 회계 감사
- 장부가 증언한다
- 카드사 앱이 조용히 말을 건다

주의할 점:

- 사용자의 상황을 과도하게 비하하지 않는다.
- 위험 상태는 웃기게 말하되, 실제 위험을 흐리지 않는다.
- 익월 유동성이 마이너스이면 가족에게 미안한 상황, 적자 보전 요청 가능성, 보정 필요성을 분명히 드러낸다.
- 카드 결제일이 가까운 경우에는 결제 압박을 읽을 수 있어야 한다.
- 카드 한도 사용률 문구는 본인 카드와 가족카드의 결합 부담을 혼동하지 않는다.

## 카테고리 문구

소비 분류 라벨과 통계 톤은 `categories.py`와 관련 YAML 문구를 함께 본다.

현재 주요 방향:

- 미분류
- 꼭 써야 했을까
- 안 썼으면 큰일
- 최소한의 품위유지비

라벨은 UI에 직접 보이므로 너무 길게 만들지 않는다. 농담을 넣더라도 드롭다운과 모바일 카드에서 잘리지 않는 길이를 우선한다.

## 선택 정책

운영에서는 같은 분기 안에서도 요청마다 다른 문구가 나올 수 있다.

테스트에서는 `choose_message(..., seed=...)`를 사용해 결과를 고정할 수 있다.

기존 호환용 `stable_choice()`는 남아 있지만, 새 Judgment 문구는 `judgment_message()`를 통해 YAML pool에서 고르는 것을 원칙으로 한다.

## 테스트 지침

문구 수정 후 최소한 다음을 확인한다.

- 본인 앱, 청구 공유, 가족카드 공유 문구 pool이 섞이지 않는다.
- `/share/claim` 문구에 정확한 개인 카드 총액이 들어가지 않는다.
- `/share/claim` 문구에 정확한 현금흐름 금액이 들어가지 않는다.
- `/share/claim` 문구에 `questionable` 같은 내부 용어가 직접 노출되지 않는다.
- `family_card` 문구에 `오빠`가 등장하지 않는다.
- 가족카드 문구가 여동생을 모욕하거나 갚지 않을 사람처럼 몰아가지 않는다.
- 테스트에서 랜덤 문구 선택이 flaky하지 않다.

권장 검증 명령:

```bash
pytest backend/tests
```

프론트엔드 표시 방식까지 건드린 경우에는 추가로 다음을 확인한다.

```bash
npm --prefix frontend run build
```

## 문구 작성 예시

좋은 방향:

- "장부를 얼핏 보니, 이번 달은 병원과 약국이 자주 등장했습니다. 아들이 생각보다 자주 고장 났습니다."
- "가족카드 사용이 활발합니다. 신용은 공유되지 않지만 한도 부담은 명의자에게 먼저 도착합니다."
- "관리 가능한 구간입니다. 다만 결제일 달력은 계속 말을 겁니다."

피해야 할 방향:

- "카드 기록은 306,840원, 현금흐름은 68,776원입니다."
- "questionable 지출 비율이 높습니다."
- "오빠가 불안합니다."
- "안 갚을 것 같습니다."
- "망했습니다."

핵심은 농담이 아니라 판단이다. 농담은 판단을 더 잘 읽히게 만드는 조미료로만 쓴다.
