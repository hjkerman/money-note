# 아키텍처

`money-note`는 개인 신용카드 생활에 맞춰 당월 기록, 결제, 현금흐름, 가족 공유를 웹에서 조작하기 위한 1인용 가계부 시스템이다.

## 기본 원칙

- 서버 DB를 원본 데이터로 사용한다.
- 가족 공유 화면은 청구/가족카드 읽기 전용만 제공하며, 네 자리 공유 PIN을 설정할 수 있다.
- 공개 화면에는 계좌번호, 송금 링크, 개인정보를 넣지 않는다.
- 유머와 평가는 `judgment` 모듈에 모아둔다.

## 구성

- 백엔드: FastAPI + SQLite
- 프론트엔드: Vite + React + TypeScript
- 배포: Docker Compose
- 모바일 앱: 웹 축소판이 아니라 빠른 입력, 현금흐름, 당월 내역, 정산, 상태 확인에 집중하는 별도 클라이언트다. 기준 화면은 [모바일 앱 설계](mobile-design.md)에 둔다.
- 모바일 앱은 서버 DB를 원본으로 사용하므로 앱 시작 시 서버에 연결할 수 없으면 종료 안내를 표시한다.

## 데이터 흐름

1. 웹 앱은 API를 통해 DB를 조회/수정한다.
2. 추가/삭제/확인 같은 조작은 즉시 서버에 저장된다.
3. 분류 변경도 드롭다운 조작 즉시 서버에 저장된다.
4. 필요하면 장부 운용 데이터 전체와 비민감 운영 설정을 JSON snapshot으로 내보내거나 복원한다.
5. 변경 API 요청은 요청 본문 없이 감사 로그에 남고, 본체 웹의 `관리 로그`에서 조회·초기화한다.

## 주요 화면 구조

- `요약 / 인사이트`: 카드대금, 고정지출, 동결자산, 유동성 등 계산값과 장부 전체 변화에 반응하는 예산심사위원회 한 줄 평
- `당월`: 당월 지출, 청구, 가족카드
- `고정지출`: 현금성 고정지출과 카드 정기결제
- `동결`: 사지 말지 보류한 임시 항목. 실제 지출은 당월 지출에 직접 기록하고 동결 항목은 삭제
- `현금흐름`: 현금 입출금 기록
- `이번달 결제`: 직전월 사용내역의 즉시결제, 일부결제와 당월 결제금액 기록
- `통계 보기`: 소비 통계와 월별 기록을 함께 표시
- 월별 기록: 구조화된 `current + archive` 기록을 `YYYY-MM` 단위로 조회한다.

주요 조작 화면의 테이블은 `panel header -> input form -> table` 구조를 따른다. 월말에 기록이 길어져도 입력창을 찾기 위해 맨 아래까지 내려가지 않도록 하기 위한 구조다.

카드 할인 혜택 정책은 사용월 기준이며 본인회원 카드와 가족카드를 분리한다. 월 정책이 `disabled`이면 할인액은 계산상 0원이다. 그 외에는 모든 카드 지출에 기본 1.2% 할인을 적용한다. `discount_override=1`인 항목은 개별 예외로 보며, 저장된 할인액이 0원이면 할인 제외로 취급한다. 기존에 저장된 할인 이벤트와 청구 할인액은 보존한다.

## 카드 결제 관리

- 원래 사용금액은 수정하지 않는다.
- 사용내역에는 월마감 후에도 유지되는 `payment_key`를 부여한다.
- 남은 결제금액은 `원래 금액 - 즉시결제 누적 - 할인액`으로 계산한다.
- 즉시결제는 현금흐름 출금을 자동 생성하지만, 할인액은 현금흐름을 만들지 않는다.
- 자동 배분은 직전월 사용내역을 1일부터 말일까지 정렬하고 현재 유동성 안에서 앞 항목부터 배분한다. 마지막 항목에는 일부 금액만 배분할 수 있다.
- 사용자는 항목 선택과 이번 처리액을 직접 수정할 수 있다.
- 카드사가 월말 이후 매입한 전월 항목은 `전월 매입 지연 보정`에서 추가하며 이번달 결제 대상에 즉시 포함한다. 과거 기록은 삭제하지 않고, 환급이 생기면 현금흐름 입금으로 기록한다.
- `교통` 문자열이 포함된 적요는 교통비로 표시하며, 사용월 기준 익월 결제 대상에 포함한다.
- 카드 사용내역은 이번 달 일부/전액 처리 또는 다음 달 이월 중 하나를 선택할 수 있다.
- 이월 시 원본 장부 행을 현재 월 사용내역 맨 앞으로 실제 이동하고, 날짜 표시는 비우며 적요 앞에 `[이월]`을 붙인다. 원본 상태는 별도 이월 테이블에 보관해 같은 달에는 복원할 수 있다.
- 이월 선택과 취소는 결제일인 14일까지 가능하며, 15일부터는 이월이 확정되어 원본 상태로 되돌릴 수 없다.
- 즉시결제는 결제일인 익월 14일까지 가능하다.

청구와 가족카드는 가족에게 돌려받을 금액이므로 내 소비가 아니라 회수 예정 금액으로 취급한다. 청구 탭 금액은 청구 탭 표시 합계와 공유 청구서의 실청구액에만 반영한다. `당월` 큰 탭 합계와 소비 통계, `next_month_liquidity` 계산에는 넣지 않는다.
- 14일이 지난 뒤 남은 결제금액은 정규결제 완료로 의제하며, 유동성은 자동 수정하지 않고 사용자의 수동 보정을 요구한다.

결제 심사 기준은 해당 결제월 현금흐름에서 `이달 기준 수입`으로 표시한 입금 합계다. 이달 기준 수입이 없으면 변경 가능한 `base_next_month_liquidity` 설정값, 즉 기본 예정 수입을 fallback 심사 기준액으로 사용한다.

## 인증 흐름

기본 인증은 서버가 발급하는 세션이다.

- 웹 브라우저: `money_note_session` HttpOnly cookie를 사용한다.
- cookie를 쓰기 어려운 클라이언트: 로그인 응답의 `session_token`을 저장하고 `Authorization: Bearer ...` 헤더를 함께 보낼 수 있다.
- 모바일 앱: `POST /api/auth/mobile-login`에서 웹 cookie 없이 장기 Bearer 토큰만 발급받는다. 유효기간은 `MONEY_NOTE_MOBILE_SESSION_DAYS`가 관리한다.

가족 공유 페이지는 본체 로그인과 분리된 `share_sessions` 장기 세션을 사용한다. 새 DB는 기본 PIN `0000`으로 잠기며, 카카오톡 인앱 브라우저를 포함한 각 브라우저에서 한 번 PIN을 입력해야 한다. 본체 사용자는 기본 PIN을 바꿀 때까지 로그인 후 경고를 받으며, PIN 변경 시 기존 공유 세션은 모두 종료된다.

## 판단 모듈

판단 모듈은 `backend/app/services/judgment/` 패키지다. 프론트엔드는 `/api/judgment/current` 응답을 받아 표시만 한다.

담당:

- 당월 지출 분류 라벨
- 소비 통계 문구
- 카드 한도 감시 문구
- 장부 전체 변화에 반응하는 예산심사위원회 문구
- 결제일과 미결제액을 함께 보는 파산심사위원회 문구
- 청구 공유 화면 상단 문구
- 가족카드 공유 화면 상단 문구
- 어머니께 보이는 청구 공유 화면의 가계부 한 줄 평가

판단 문구는 무작위로 바뀌지 않는다. 같은 장부 상태에는 같은 문구를 보여주고, 관련 금액·날짜·건수가 바뀌면 같은 위험 등급 안에서도 다른 후보 문구를 선택한다. 상단 예산심사위원회 총평은 현금, 카드, 청구, 가족카드, 동결 중 어느 영역에든 새 항목이 추가될 때마다 문구가 순환한다.
모바일 앱은 같은 `/api/judgment/current` 응답을 재사용한다. 입력 탭에는 대표 문구를 짧게 보여주고, 상태 탭에서는 예산심사위원회, 카드 한도 감시, 파산심사위원회를 함께 보여준다. 모바일 클라이언트는 별도 판단을 생성하지 않는다.

분류 체계:

- `essential`: 안 썼으면 큰일 났을 돈
- `questionable`: 꼭 써야 했을까...?
- `dignity`: 최소한의 품위유지비
- `NULL`: 미분류

## 적요 구조

당월 지출은 앱에서 `사용처`와 `사용항목`으로 나눠 입력할 수 있다.
웹의 당월 지출과 월별 기록 표는 `사용처`, `사용항목`, `분류`를 서로 다른 열로 표시하고, 카드 정기결제 표도 `사용처`, `사용항목`을 분리해 표시한다.
DB에는 두 필드를 보존하고, `title`에는 사람이 한눈에 읽기 쉬운 대표 적요를 저장한다.

## 월마감

월마감은 자동 실행하지 않는다. 달력상 새 달인데 이전 월 기록이 `current`에 남아 있으면 첫 화면에서 월마감 검토를 요청한다. 사용자가 월말에 장부를 먼저 닫는 운용을 위해 매월 27일부터 현재 달 조기 마감을 허용하며, 반드시 별도 확인을 요구한다.

월마감은 `current`에 남은 일반 지출 중 가장 오래된 월 하나만 `archive`로 복사하고 current에서 삭제한다. 새 달 기록을 먼저 입력한 뒤 월마감해도 새 달 기록은 그대로 남는다. 이미 마감된 월 날짜로 일반 지출을 추가하면 다음 달 장부에 섞지 않고 `archive`에 바로 추가한다.

카드 정기결제와 같은 planned 항목은 current에 남는다.
청구와 가족카드는 월마감 및 카드 결제 주기와 독립된 전달용 임시 큐다. 조회·추가·공유·일괄 완료는 미마감 장부가 아니라 달력상 현재 연-월을 기준으로 한다. 사용자가 `일괄 처리 완료`를 누르면 해당 월·타입의 현재 항목을 삭제한다.

청구는 회수 예정 금액이므로 소비 분류 대상에서 제외한다. 본인용 청구 입력 표와 통계에는 자동 분류를 표시하지 않는다.

## 감사 로그

웹 중심 감사 로그는 `POST`, `PATCH`, `DELETE` API의 사용자, 방식, 경로, 결과 코드만 기록한다. 요청 본문, 비밀번호, 세션 토큰과 SQL 전문은 기록하지 않는다. 감사 로그 장애는 실제 가계부 조작을 실패시키지 않으며, 사용자는 관리 화면에서 로그 전체를 초기화할 수 있다.
