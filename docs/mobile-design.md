# 모바일 앱 설계

Figma 원본:

- [Money Note Mobile App](https://www.figma.com/design/ys5W1k5bwO7YQi7sWWTZK3)

이 문서는 모바일 앱 구현 시 기준으로 삼을 화면 구조와 제품 원칙을 정리한다.

## 제품 방향

Money Note 모바일 앱은 웹 앱의 축소판이 아니다.

모바일 앱의 목적은 돈을 쓴 직후 5~10초 안에 기록하고, 현금흐름을 빠르게 보정하며, 가족 관련 정산 상태를 확인하는 것이다.

우선순위:

1. 현재 상태와 최근 입력 확인
2. 카드 알림 후보 처리와 수동 입력
3. 청구와 가족카드 정산
4. 현금흐름 관리
5. 낮은 빈도의 운영 설정

복잡한 분석, 관리 로그, 복원 같은 기능은 일상 흐름을 방해하지 않도록 설정 화면에 둔다.
모바일 앱은 서버 DB를 원본으로 사용하므로 앱 시작 시 서버에 닿지 않으면 종료 안내를 보여준다.

## 문구 원칙

모든 사용자 노출 문구는 한국어로 작성한다.

예시:

- `청구`
- `가족카드`
- `상태`
- `익월 유동성`
- `스냅샷 다운로드`

영어 기능명을 그대로 노출하지 않는다.

## 디자인 원칙

- 카드형 화면 구조를 사용한다.
- 표를 사용하지 않는다.
- 리스트와 카드 중심으로 표시한다.
- 한 손 조작을 우선한다.
- 금액은 크게 표시한다.
- 정보 밀도는 높게 유지하되 입력 흐름은 짧게 둔다.
- 애니메이션은 최소화한다.
- 장식보다 실용성을 우선한다.

## 주 검증 기기

우선 검증 타깃은 Galaxy S23 FE / One UI 8.5다.

이 기기 기준으로 다음을 우선 확인한다.

- 한 손 입력이 불편하지 않은지
- 하단 탭과 입력 버튼이 시스템 내비게이션과 겹치지 않는지
- 알림 접근 권한 화면으로 정상 이동하는지
- 우리카드 승인 알림이 본인카드/가족카드 후보함으로 잘 분류되는지
- `최근 우리카드 알림`에서 실사용 파싱 결과를 확인할 수 있는지
- 설정 맨 아래 `Experimental Data`에서 교통·통행 원문을 실사용 흐름과 분리해 확인할 수 있는지
- 큰 글꼴 설정에서 금액과 버튼 문구가 깨지지 않는지

## 하단 탭

하단 탭은 실제 사용 빈도에 맞춰 다섯 개를 둔다.

- 홈
- 정산
- 입력
- 현금
- 설정

좌우 스와이프와 하단 탭 선택은 같은 순서로 이동한다.

## 홈 화면

앱 실행 시 기본 화면이다.

상단에는 현재 월과 다음 요약값을 표시한다.

- 카드대금
- 월 지출
- 익월 유동성
- 동결

그 아래에는 서버가 완성한 다음 Judgment 문구를 그대로 표시한다.

- 예산심사위원회
- 카드 한도 감시
- 파산심사위원회

홈의 `알림에서 가져오기`는 우리카드 승인 후보함으로 이동한다. 최근 입력 10건과 입력 탭으로 이동하는 `내역 수동 입력` 버튼도 둔다. 홈에서 별도 금액 계산이나 Judgment 생성을 하지 않는다.

## 입력 화면

현재 월 카드 지출을 등록하고 원장 전체를 확인하는 화면이다.

입력 항목:

- 사용 일자
- 사용처
- 사용항목
- 분류
- 금액
- 할인 적용 여부

- 지출 추가

목표는 사용자가 10초 안에 지출을 기록하는 것이다.

원칙:

- 입력 폼 바로 아래에서 당월 지출 전체를 조회한다.
- 표를 쓰지 않는다.
- 사용처, 사용항목, 금액, 사용일을 카드 안에 표시한다.
- 각 행의 분류, 할인 적용 여부, 실결제액을 수정하고 삭제할 수 있다.
- 교통, 하이패스, 통행료처럼 결제 정책상 눈에 띄어야 하는 항목은 작은 배지로 표시할 수 있다.

## 현금 화면

현금흐름 관리 전용 화면이다.

상단에는 현재 유동성과 익월 유동성을 표시한다.

현금 입력 카드:

- 입금/출금 전환
- 내용
- 금액
- 이달 기준 수입 여부

현금흐름 항목은 카드 형태로 표시한다.

항목 카드에는 다음 정보를 둔다.

- 발생일
- 내용
- 입금/출금 금액
- 이달 기준 수입 표시
- 삭제 버튼

표 형태는 사용하지 않는다.

## 정산 화면

상단 카드 전환으로 `청구`와 `가족카드`를 오간다.

청구 화면:

- 원금 합계
- 실청구 합계
- 공유하기
- 일괄 처리 완료
- 항목별 원금, 할인, 실청구, 삭제
- 등록 전 할인 적용 여부 선택

가족카드 화면:

- 원금 합계
- 실결제 합계
- 가족카드 혜택 상태
- 공유하기
- 일괄 처리 완료
- 등록 전 할인 적용 여부 선택

청구와 가족카드는 원장이 아니라 회수 예정 정보다. 따라서 모바일에서도 소비 통계나 유동성의 핵심 흐름처럼 보이지 않도록 분리한다.

## 설정 화면

일상 입력보다 빈도가 낮은 운영 기능을 모은 화면이다.

표시 항목:

- 카드대금
- 월 지출
- 익월 유동성
- 동결
- 동결 금액
- 현금성 고정지출
- 카드 정기결제
- 월마감
- 백업 / 복원
- 설정

각 관리 항목은 별도 세부 화면으로 들어간다. 관리 기능은 접근 가능해야 하지만 일상 입력 흐름을 방해하지 않아야 한다.

## 향후 기능 공간

모바일 앱은 다음 기능을 염두에 둔다.

- 교통 알림 후보 생성
- 통행료 알림 후보 생성

우리카드 승인 후보 기능은 현재 실사용 중이다. 교통과 통행료는 원문을 관측한 뒤 별도 승격 여부를 결정한다.

현재 Android 구현 원칙:

- 알림 리스너는 하나만 유지하고 패키지명으로 출처를 구분한다.
- 우리카드 `com.wooricard.smartapp`의 일시불 승인 알림은 로컬 후보함에 저장한다.
- 알림 수신 즉시 서버 등록은 하지 않는다.
- 사용자가 후보를 확인하고 `등록`을 눌렀을 때만 기존 API를 호출한다.
- 우리카드의 할부 승인, 파싱 실패, 광고/혜택 안내는 후보로 만들지 않고 본인카드 원문 로그에 남긴다.
- 고속도로 통행료+ `com.ex.hipass_app` 알림은 하이패스 원문 로그에만 남긴다.
- 모바일티머니 `com.lgt.tmoney` 알림은 삼성월렛 모바일티머니의 결제 금액·잔액을 관측하기 위한 교통 원문 로그에만 남긴다.
- 출처별 원문 로그는 각각 최근 30건만 유지한다.
- 우리카드 원문은 실사용 메뉴 `설정 -> 최근 우리카드 알림`에서 단독으로 확인한다.
- 설정 맨 아래 `Experimental Data`는 `교통 / 통행` 순서로 모바일티머니와 고속도로통행료+ 관측 자료를 분리한다.
- 각 원문 카드는 `packageName`, `title`, `text`, `bigText`, `subText`, `textLines`, `rawText`, 알림 key와 게시 시각 등을 표시하고 개별 공유할 수 있다.

하이패스 알림의 공개된 본문 규격은 아직 없다. 원문 표본이 충분히 쌓이면 우리카드 파서와 분리된 하이패스 파서를 추가한다. 파싱 결과는 다른 카드 후보와 같은 방식으로 사용자가 날짜, 금액, 본인 사용 또는 청구 사용을 확인한 뒤에만 서버에 등록한다.

사용자의 실기기에서는 모바일티머니 앱이 삼성월렛 모바일티머니의 결제 금액과 잔액 알림을 제공하는 것을 확인했다. 현재는 모바일티머니 원문만 수집한다. 원문 표본이 충분히 쌓이면 별도 모바일티머니 파서를 추가하고, 날짜·금액·본인 사용 또는 청구 사용을 확인하는 등록 후보로 만든다. 자동 서버 등록은 하지 않는다. 충분히 안정화된 뒤에만 실사용 기능으로 승격해 우리카드 알림 관측 화면과 통합한다.
